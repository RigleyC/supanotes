# Note Title From Nodes Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make every note title (backend API, search, AI listings, and Flutter UI) come from the first text-bearing `note_nodes` row instead of regex on a markdown/string `content` snapshot. The `KeepFirstLineAsTitleReaction` (which enforces H1 styling of the first line in the editor) is intentionally preserved — it serves a different purpose than title derivation and is still required for the Apple Notes-style first-line-as-title UX.

**Architecture:** `note_nodes` is the source of truth for note content. The title is the `data->>'text'` of the first non-deleted node (by `position`) whose text is non-empty, regardless of node type. The backend SQL queries (`notes.sql`, `search.sql`, `ai.sql`) use a correlated subselect instead of `regexp_replace(split_part(...))`. The Flutter `NotesDao` joins the same subselect and carries a `title` field through `NoteQueryResult` into `NoteModel`, replacing the `deriveNoteTitle(content)` getter. `deriveNoteTitle` (Dart) is deleted. The `KeepFirstLineAsTitleReaction` stays — it is the SuperEditor mechanism that keeps the first line visually H1 during editing (a separate concern from where the listed title comes from). The Go `DeriveTitle(content)` helper stays for now — it is only used on the AI/markdown-export path (sanctioned as "md for AI"); removing it is Frente 2 (out of scope).

**Tech Stack:** Go, SQLC, PostgreSQL, Flutter, Drift (SQLite), Riverpod, SuperEditor

## Canonical SQL subselect

Every backend title expression becomes this (alias `AS title`):

```sql
COALESCE((
  SELECT (nn.data->>'text')::text
  FROM note_nodes nn
  WHERE nn.note_id = n.id
    AND nn.deleted_at IS NULL
    AND nn.data->>'text' IS NOT NULL
    AND nn.data->>'text' <> ''
  ORDER BY nn.position ASC
  LIMIT 1
), 'Untitled')::text AS title
```

The Flutter DAO uses the same subselect **without** `COALESCE` (returns nullable), and applies the `'Sem título'` fallback in Dart:

```sql
(SELECT nn.data->>'text'
 FROM note_nodes nn
 WHERE nn.note_id = n.id
   AND nn.deleted_at IS NULL
   AND nn.data->>'text' IS NOT NULL
   AND nn.data->>'text' <> ''
 ORDER BY nn.position ASC
 LIMIT 1) AS title
```

## Out of scope (Frente 2 — follow-up plan)

These are intentionally **not** in this plan. They belong to a separate "nodes end-to-end write path" effort:

- Backend `CreateNote`/`UpdateNote`/`ApplyOrganization` still accept a markdown `content` string and call `ParseMarkdownToNodes` to derive nodes. Canonicalizing the write path to accept nodes directly is a larger schema/API change.
- Backend `auth/service.go` seed still uses `"# Rascunho\n\n..."` markdown.
- Go `notes.DeriveTitle(content)` (in `backend/internal/notes/service.go:411`) and its callers in `backend/internal/agent/context.go` and `backend/internal/agent/tools/notes_tools.go` still derive a title from a markdown string. These operate on the AI/markdown-export path, which is sanctioned as "md for AI". They can be replaced with the query-row `Title` field in a follow-up that gives the agent context access to node-derived titles without re-deriving from content.
- `notes.content` (the plain-text/markdown snapshot) is still synced and indexed by FTS. Decanonizing it is part of Frente 2.
