# Plano: Colaboração de Notas V0 por Operações REST

## Objetivo

Substituir integralmente o sync de notas baseado em Yjs por um fluxo pequeno,
auditável e testável de operações versionadas. A V0 deve provar, em dois
clientes Flutter e sem WebSocket, que edição concorrente, edição offline,
reconexão e reenvio não perdem nem embaralham texto.

Este é um reset intencional do conteúdo de produção. Não há migração de
YDoc/Yjs, `note_yjs_states` ou dados existentes. O backup fora do banco é a
fonte de recuperação caso a experiência V0 não seja aprovada.

## Decisões de arquitetura

1. `MutableDocument`/`super_editor` continua sendo somente a superfície de
   edição. Ele nunca é salvo como fonte de verdade por meio de um snapshot
   inteiro após cada alteração.
2. O servidor é a autoridade de ordenação: cada nota possui uma revisão
   inteira, crescente e sem lacunas.
3. Texto rico é representado pelo formato Quill Delta (`insert`, `delete`,
   `retain`, `attributes`). A transformação OT acontece apenas dentro de um
   bloco de texto.
4. Estrutura da nota usa IDs estáveis, nunca posições numéricas:
   `createBlock`, `deleteBlock` e `moveBlock(afterBlockId)`.
5. REST é o único transporte da V0. Enquanto a nota estiver aberta, o cliente
   busca operações novas por polling leve; não há WebSocket, cursores,
   presença, leases ou `NOTIFY`.
6. Cada mudança local é gravada no SQLite no outbox antes de depender da rede.
   Uma queda, encerramento ou timeout nunca descarta uma operação local.
7. A V0 não tem dois caminhos de escrita para nota: nenhum YDoc, projeção Yjs,
   `SyncService.push()` de conteúdo, `tasks` write-through ou substituição de
   snapshot pode concorrer com o log de operações.
8. O backend continua em Go. A dependência `go-quilljs-delta` só pode ser
   adotada depois do gate de compatibilidade com `dart_quill_delta`; não será
   escrito um transformador OT próprio.

## Escopo funcional V0

### Incluído

- Criar, abrir e compartilhar nota entre dois usuários com permissão `edit`.
- Parágrafos, quebras de bloco, apagar, inserir e mover blocos.
- Atributos inline da toolbar: bold, italic e strike-through.
- Tipos de bloco já expostos pela toolbar: parágrafo, H1/H2/H3, citação,
  lista ordenada, lista não ordenada e divisor.
- Trabalho offline, fila local persistida, reenvio idempotente e polling de
  reconciliação.
- Atualizar título/excerpt da listagem a partir do documento canônico.

### Explicitamente fora da V0

- WebSocket, cursores, presença, chat e colaboração em tempo real por frame.
- Anexos, imagens, menções, links enriquecidos, IA, embeddings e busca por
  conteúdo rico.
- Operações colaborativas de task, recorrência, notificações e
  `task_completions`. O componente e a UI de task permanecem no repositório,
  mas o botão de task deve ser temporariamente indisponível no editor V0 para
  não criar dados sem protocolo canônico.
- Migração ou leitura do conteúdo Yjs antigo.

> Confirmação necessária antes da implementação: esta interpretação considera
> que `task` fica fora da V0 e que os exemplos `H1` e `bold` devem funcionar.
> Se a frase "exceto task, h1, bold" pretendia excluir também formatação e
> headings, o adaptador e a matriz de testes devem ser reduzidos antes do
> início.

## Modelo canônico no servidor

O conteúdo canônico é um JSONB de blocos, porque a V0 não precisa de consultas
SQL por bloco. O log é a fonte de auditoria e reconciliação; o snapshot reduz
o tempo de abertura.

```json
{
  "schemaVersion": 1,
  "blocks": [
    {
      "id": "01J...",
      "type": "paragraph",
      "delta": [{"insert": "Treino ", "attributes": {"bold": true}}, {"insert": "hoje"}],
      "metadata": {}
    }
  ]
}
```

### Tabelas PostgreSQL

`notes` permanece como a entidade de lista, posse e compartilhamento. Ela
recebe campos novos e deixa de usar `content` como fonte de verdade:

```sql
ALTER TABLE notes
  ADD COLUMN revision BIGINT NOT NULL DEFAULT 0,
  ADD COLUMN document JSONB NOT NULL DEFAULT '{"schemaVersion":1,"blocks":[]}'::jsonb,
  ADD COLUMN snapshot_revision BIGINT NOT NULL DEFAULT 0;
```

- `content` e `excerpt` são projeções de leitura para lista, busca futura e
  compatibilidade da UI. O serviço de operações os atualiza na mesma
  transação; clientes nunca escrevem esses campos.
- `revision` é a única versão de reconciliação da nota.

```sql
CREATE TABLE note_operations (
  note_id UUID NOT NULL REFERENCES notes(id) ON DELETE CASCADE,
  revision BIGINT NOT NULL,
  operation_id UUID NOT NULL,
  actor_id UUID NOT NULL REFERENCES users(id),
  base_revision BIGINT NOT NULL,
  kind TEXT NOT NULL,
  block_id TEXT,
  payload JSONB NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (note_id, revision),
  UNIQUE (note_id, operation_id),
  CHECK (revision > 0),
  CHECK (base_revision >= 0)
);

CREATE INDEX idx_note_operations_since_revision
  ON note_operations(note_id, revision);
```

Não criar tabelas `note_blocks`, `note_yjs_states`, `note_yjs_updates`,
state vector, tombstones CRDT ou snapshots duplicados nesta V0. O snapshot em
`notes.document` mais o log é suficiente.

### Contrato de operação

```json
{
  "operationId": "uuid",
  "baseRevision": 12,
  "kind": "text_delta",
  "blockId": "01J...",
  "payload": {"ops": [{"retain": 4}, {"insert": " novo"}]}
}
```

Tipos permitidos:

| Tipo | Payload | Regra de conciliação |
| --- | --- | --- |
| `text_delta` | Delta Quill para um bloco `TextNode` | OT contra deltas posteriores do mesmo bloco |
| `create_block` | bloco completo e `afterBlockId` | IDs distintos sempre coexistem; mesma âncora é ordenada por revisão/operação |
| `delete_block` | `blockId` | delete vence; operação de texto posterior no bloco removido é rejeitada como `BLOCK_DELETED` |
| `move_block` | `blockId`, `afterBlockId` | âncora por ID; se a âncora foi apagada, inserir no fim e devolver o resultado canônico |
| `set_block_type` | tipo permitido | última operação ordenada pelo servidor vence |

Não haverá `update_block` contendo um nó inteiro. Essa operação seria uma
substituição escondida de snapshot e reintroduziria perda de atributos.

## Protocolo REST

### Abrir/recarregar

`GET /api/v1/notes/:noteId/document`

Resposta: `noteId`, `revision`, `document`, `serverTime`. O cliente grava o
snapshot local e passa a conhecer a revisão base.

`GET /api/v1/notes/:noteId/operations?afterRevision=:revision`

Resposta ordenada por revisão. Enquanto o editor estiver visível, polling a
cada 1 segundo apenas depois de haver atividade remota potencial; ao voltar
ao foreground, ao recuperar conectividade e antes de enviar um lote, polling
imediato. Nenhum polling global de todas as notas.

### Enviar lote

`POST /api/v1/notes/:noteId/operations:sync`

```json
{
  "knownRevision": 12,
  "operations": [ ... ],
  "clientId": "uuid-persistido-na-instalacao"
}
```

O servidor, em uma única transação e com `SELECT ... FOR UPDATE` na nota:

1. autoriza o usuário; somente dono ou `note_shares.permission = 'edit'`;
2. deduplica por `(note_id, operation_id)` e retorna o ack anterior;
3. lê o log de `baseRevision + 1` até a revisão atual;
4. para `text_delta`, transforma a operação contra alterações posteriores no
   mesmo bloco, com desempate determinístico por `operation_id`;
5. valida a operação transformada contra `notes.document`;
6. aplica a operação, incrementa a revisão, projeta `content`/`excerpt` e
   insere uma linha em `note_operations`;
7. devolve as operações canônicas aceitas, a revisão final e quaisquer
   operações remotas que o cliente ainda não tinha.

Resposta de erro é estruturada: `INVALID_DELTA`, `BLOCK_DELETED`,
`INVALID_ANCHOR`, `NOTE_DELETED`, `FORBIDDEN` ou `SCHEMA_VERSION_UNSUPPORTED`.
Erro não remove o item do outbox. O cliente mantém uma cópia exportável do
item rejeitado e mostra erro recuperável; ele nunca descarta edição silenciosa.

## Reconciliação no cliente

O cliente mantém três estados separados para cada nota aberta:

1. **confirmado**: snapshot na última revisão confirmada pelo servidor;
2. **pendente**: operações do outbox, na ordem de criação;
3. **visível**: `confirmado + pendente`, aplicado no `MutableDocument`.

Para operação local:

1. o adaptador do editor cria uma operação mínima;
2. grava snapshot/outbox no Drift em uma transação;
3. aplica otimisticamente ao estado visível;
4. agenda `sync` com debounce de 250 ms, e força flush em pause/dispose.

Para operação remota:

1. aplicar a operação ao estado confirmado;
2. transformar cada `text_delta` pendente contra ela, em ordem;
3. reconstruir apenas os blocos afetados no editor, preservando seleção com
   `transformPosition`; não substituir o documento inteiro;
4. persistir o novo confirmado e as operações pendentes rebaseadas no Drift.

Após ack, remover somente os `operationId` confirmados. `operationId` é
gerado uma vez e nunca é alterado em retry.

## Persistência local Drift

Adicionar tabelas locais, sem reutilizar `note_yjs_states`:

```text
local_note_documents
  note_id PK, revision, document_json, updated_at

pending_note_operations
  operation_id PK, note_id, base_revision, ordinal,
  kind, block_id nullable, payload_json,
  created_at, last_attempt_at nullable, attempt_count

note_sync_errors
  operation_id PK, note_id, error_code, message, payload_json, created_at
```

- Índice em `pending_note_operations(note_id, ordinal)`.
- O `ordinal` define a ordem local, independente de relógio.
- Uma operação só sai do outbox após confirmação explícita do servidor.
- A operação e a mudança de snapshot local são atômicas em uma transação
  Drift. Não usar isolate para abrir uma nota pequena; parse e reconstrução
  devem ser avaliados por tamanho e movidos para isolate somente quando a
  medição justificar.

## Adaptador do super_editor

Criar `NoteOperationAdapter` que observa `DocumentChangeLog` e traduz eventos
para operações do protocolo. Ele substitui `EditorDocumentSyncManager` e
`YjsDocEditorBridge`.

- `NodeInsertedEvent` -> `create_block`.
- `NodeRemovedEvent` -> `delete_block`.
- `NodeMovedEvent` -> `move_block` por âncora estável.
- Alteração de `TextNode` -> Delta entre a versão anterior confirmada/visível
  daquele bloco e a versão nova, incluindo atributos.
- Mudança de heading/lista/citação -> `set_block_type`.
- Divisor -> `create_block` de tipo `divider` sem Delta.

O adaptador conserva um espelho serializado por bloco para calcular o Delta;
não usa `diff_match_patch` de texto simples, pois isso perderia atributos.
O gate de compatibilidade define se a conversão deve usar o histórico de
`EditRequest` do Super Editor ou um diff de Delta rico.

## Gate obrigatório: compatibilidade Delta

Antes de qualquer remoção ou reset, criar um pacote de teste isolado:

- Dart: `dart_quill_delta` fixado em versão explícita, não o pacote antigo
  `quill_delta` incompatível com Dart 3.
- Go: `github.com/fmpwizard/go-quilljs-delta/delta`, versão fixada no
  `go.mod`.
- Fixtures JSON compartilhadas entre Dart e Go para `compose`, `transform`,
  `transformPosition`, inserção simultânea, deletes sobrepostos, atributos
  simultâneos e atributos removidos.
- Fuzz determinístico com seed persistida e no mínimo 10.000 pares de deltas.
- Simulação de dois clientes offline, múltiplos pendentes, retry duplicado,
  chegada fora de ordem e reinício do processo.

**Stop condition:** qualquer divergência entre Dart e Go, perda de atributo ou
operação inválida que não tenha erro explícito interrompe o projeto. Nesse
caso não apagar Yjs nem resetar produção; reavaliar o motor OT antes de mudar
arquitetura.

## Remoção do fluxo legado

Após o gate passar e a V0 operar em ambiente de teste, remover apenas o que
pertence ao sync Yjs. Preservar `super_editor`, toolbar, task widget e os
demais recursos de UI fora do escopo.

### Flutter

- Remover `yjs_dart` de `pubspec.yaml`, `packages/yjs_dart/` e o lockfile.
- Remover `lib/core/sync/yjs_sync_manager.dart` e seu provider.
- Remover `yjs_note_schema.dart`, `yjs_node_codec.dart`,
  `yjs_doc_editor_bridge.dart`, `EditorDocumentSyncManager` e testes Yjs.
- Remover tabelas/DAOs Drift de estado Yjs e qualquer cache de state vector.
- Retirar `YjsSyncManager` de `SyncService`, DI e ciclo de vida de editor.
- Fazer `SyncService` deixar de escrever/puxar conteúdo de notas; mantê-lo
  somente para recursos que sobreviverem ao reset (preferências, tags e
  contextos, se ainda forem mantidos). `NoteOperationsSyncService` é o único
  dono do conteúdo de nota.

### Go

- Remover `github.com/reearth/ygo` de `backend/go.mod`.
- Remover `backend/internal/sync/ydoc_service.go`, `compactor.go`,
  `projection.go` e seus testes/inicialização.
- Remover comandos e ferramentas ligados a Yjs, incluindo `backend/cmd/squash`
  e scripts de inspeção exclusivos de `note_yjs_*`.
- Remover `note_yjs_states`, `note_yjs_updates`, `note_ws_leases`, queries
  SQLC, payloads REST e caminhos de compatibilidade `content_fixed`.
- Substituir o push/pull genérico de notas por handlers específicos do
  protocolo de operações; nenhum endpoint legado deve aceitar `content` para
  alterar uma nota.
- Regenerar SQLC e apagar código gerado que não tiver mais query de origem.

## Reset e rollout de produção

O reset é destrutivo e só acontece depois do gate, testes de integração e uma
confirmação operacional explícita. Ele não é uma migration normal executada
silenciosamente no deploy.

1. Colocar a API em manutenção e impedir novas escritas.
2. Criar `pg_dump` completo, criptografado, armazenado fora do Fly; restaurar
   esse dump em banco temporário e verificar contagens.
3. Criar um novo baseline de migrations para o esquema que sobrevive. Excluir
   migrations e tabelas exclusivamente Yjs; não deixar esquema morto como
   "compatibilidade".
4. Executar um comando administrativo versionado que trunca todas as tabelas
   de dados de produto, incluindo notas, shares, tasks, completions, tags,
   contexts, anexos, embeddings, rotinas, memórias, mensagens, preferências,
   tokens de dispositivo e logs derivados. A política para `users` e refresh
   tokens é decidida antes de executar: padrão recomendado é preservar
   identidades/autenticação e apagar todo conteúdo; reset total apaga também
   usuários e exige novo login/cadastro.
5. Limpar SQLite local no primeiro start da versão nova por `schemaVersion`;
   o app nunca tenta enviar outbox Yjs antigo.
6. Aplicar migrations novas, subir a API, criar duas contas/notas de teste e
   executar o roteiro de aceitação antes de liberar o uso normal.

## Etapas de implementação

1. **Contrato e harness**: definir schemas JSON, fixtures e testes cruzados
   Dart/Go. Nenhum código de produção removido.
2. **Núcleo Go**: pacote de domínio `noteoperations`, validador, transformador
   Delta, transação por nota, log e endpoints REST. Testes de repositório com
   PostgreSQL real.
3. **Armazenamento Flutter**: novas tabelas Drift, outbox, cliente REST e
   testes de retry/idempotência.
4. **Adaptador do editor**: converter `super_editor` para operações e aplicar
   operações remotas por bloco. Toolbar do escopo V0 precisa produzir e
   renderizar o mesmo Delta após reabertura.
5. **Cenários concorrentes**: dois bancos Drift + um backend de teste;
   concorrência, offline, restart, duplicação, delete/move e formatação.
6. **Troca de dono**: remover Yjs e todos os caminhos duplicados depois de os
   testes V0 passarem.
7. **Reset controlado e deploy**: seguir o procedimento de rollout acima.

## Critérios de aceite

- Dois clientes, a partir da mesma revisão, fazem 1.000 operações concorrentes
  de texto/formatação e convergem byte a byte no JSON canônico.
- Um cliente fica offline, faz ao menos 100 operações, reinicia o app e
  reconcilia sem operação faltante ou duplicada.
- Inserções na mesma posição e alterações de `bold`/`italic`/`strike` não
  perdem caracteres nem atributos.
- Nenhum fluxo de nota escreve `notes.content` diretamente fora do serviço de
  operações.
- `rg -i "yjs|ydoc|note_yjs|content_fixed|state.vector" lib backend` não
  retorna código de produção após a remoção; exceções só podem existir em
  documentação de migração/backup.
- Não há isolate no caminho de abrir uma nota V0 sem uma medição que prove sua
  necessidade.
- Teste manual em dois celulares: editar alternadamente, simultaneamente,
  offline, fechar/reabrir em ordens diferentes e comparar o documento final.

## Riscos assumidos

- A compatibilidade de Delta entre Dart e Go é a maior incerteza; o gate existe
  para impedir que ela vire uma aposta de produção.
- A V0 deliberadamente não oferece edição colaborativa de tasks. O componente
  é preservado para uma fase posterior, quando houver operações tipadas e uma
  única projeção para `tasks`/`task_completions`.
- Polling REST não é instantâneo como WebSocket, mas não compromete
  convergência. WebSocket pode reutilizar exatamente o mesmo protocolo depois,
  sem alterar armazenamento ou algoritmo.
