# 0006: Block Tree Invariants

## Status

Accepted

## Context

The note editor uses a block tree model where each document node (paragraph, task, header, list item, divider, image) is serialized into a Yjs document (YDoc). The YDoc is the single source of truth for note content (ADR 0005). The block tree must maintain certain invariants to ensure correct ordering, stable references, and consistent behavior across clients and the AI agent.

Key questions addressed:
- How are blocks identified and referenced?
- How is block ordering maintained?
- How does the AI agent refer to specific blocks?
- How does a move/reorder operation work without conflicts?

## Decision

### Task state (planned extension)

Planned for Phase 4 (see Plan 058): Task state will be stored in a dedicated `YMap("tasks")` keyed by `task-id`, rather than as node entries in `YMap("nodes")`. This separates task metadata (completion, recurrence, due dates) from document content, enabling efficient task-specific projections without scanning all nodes. The `data.completed` and `data.lastCompletedAt` fields in node metadata described below are the current approach; future migration will move this to `YMap("tasks")`.

## Decision

### Block IDs are immutable

Each block gets a UUID at creation time stored as the YMap key in the `nodes` YMap. This ID is **never changed** for the lifetime of the block. References to a block (from the AI agent, from comments, from task metadata, from YText content references) use this ID.

- The YMap `nodes` uses the block ID as the key: `nodesMap.setAttr(blockId, jsonEncode(...))`
- Block IDs are UUIDs generated by the client at creation time.
- Deleting a block removes its entry from the `nodes` YMap. Re-creating a block with the same content produces a new ID.

### Position is fractional indexing

Block ordering is represented by a `position` string stored in each block's metadata, not by YArray index. Positions are generated using the `fractional-indexing` algorithm (`GenerateKeyBetween`) which produces sortable strings between any two existing positions.

- The `position` field is stored in the block metadata JSON: `{..., position: "a0", ...}`
- Moving a block = rewriting its `position` string in the YMap entry (no YArray splice needed).
- Inserting between two blocks = `GenerateKeyBetween(prev.position, next.position)`.
- Inserting at the end = `GenerateKeyBetween(last.position, "")`.
- Inserting at the beginning = `GenerateKeyBetween("", first.position)`.
- Sorting uses lexicographic string comparison of `position` values (the fractional-indexing algorithm produces monotonically ordered strings).
- This approach avoids conflicts from concurrent YArray insertions at the same index.

### Block metadata structure

Each entry in the `nodes` YMap is a JSON string with:

```json
{
  "id": "<uuid>",
  "parentId": "<uuid or ''>",
  "position": "<fractional-index string>",
  "type": "paragraph|task|header|list_item|divider|image|attachment",
  "data": { ... type-specific fields ... },
  "createdAt": <epoch ms>
}
```

- The `id` in the JSON body must match the YMap key (enforced by the serialization layer).
- `type` determines the rendering component and projection behavior.
- `data` contains type-specific fields (text, completed, dueDate, recurrence, level, etc.).
- `parentId` supports future nesting (currently unused, always empty string).

### AI agent references blocks by ID

The AI agent (tools like `append_to_note`, `complete_task`, `move_block`) references blocks by their immutable ID. Agent operations are declarative mutations applied to the YDoc:

- `InsertParagraph(nodeId, text, afterNodeId)`: creates a new block with a fresh UUID, positions it after `afterNodeId`.
- `CompleteTask(taskId)`: reads the block from YDoc, sets `data.completed`, adds `data.lastCompletedAt`.
- `MoveBlock(nodeId, afterNodeId)`: rewrites the `position` field of `nodeId` to be after `afterNodeId`.

The agent never reconstructs the entire YDoc — it performs targeted operations on specific blocks.

## Considered Options

- **YArray for ordering**: Rejected because concurrent YArray insertions at the same index produce unpredictable merge results. Fractional indexing gives deterministic ordering independent of operation order.
- **Sequential counter for IDs**: Rejected because offline clients would generate conflicting IDs. UUIDs ensure global uniqueness without coordination.
- **Mutable block IDs (rename on merge/reparent)**: Rejected because the AI agent and external references depend on stable identifiers. Renaming would break existing references.

## Consequences

- The AI agent can safely reference blocks by ID without worrying about ID changes.
- Reordering is O(1) per block (single YMap `setAttr` for the position field) and conflict-free under CRDT merge.
- UUID generation is required for all new blocks, both from the editor UI and from the AI agent.
- Sorting the full block list requires reading all positions and doing an O(n log n) sort in the projection layer.
- The fractional-indexing helper (`GenerateKeyBetween`) must be available in both Go (backend projection) and Dart (Flutter editor).
